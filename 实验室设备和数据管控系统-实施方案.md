# 实验室设备与数据管控系统 - 实施方案

## 一、项目概述

### 1.1 项目背景

实验室是科研工作的重要场所，通常包含多个楼层、多个房间，每个房间内配备不同的实验设备。随着实验室规模的扩大和研究课题的增多，传统的人工记录方式已无法满足管理需求：

- **设备使用不透明**：谁在用、用了多久、用于什么课题，无法实时掌握
- **数据归属混乱**：实验数据产生后，难以追溯是哪台设备、哪个课题、哪个研究人员产生的
- **资源利用率低**：无法统计设备的实际使用率，导致资源浪费或排队等待
- **管理效率低**：依赖 Excel 表格，数据分散、查询困难

本系统旨在构建一个**实验室设备与数据一体化管控平台**，实现设备全生命周期管理、使用行为追踪、实验数据归档和可视化分析。

### 1.2 项目目标

1. **设备可视化管理**：直观展示每台设备的状态、位置、使用情况
2. **使用行为追踪**：记录谁在什么时间使用了哪台设备、用于哪个课题
3. **数据归属管理**：实验数据自动关联设备、项目和研究人员
4. **决策数据支撑**：通过统计图表展示设备利用率、使用趋势、项目数据分布
5. **权限分级管控**：不同角色（管理员/实验室管理员/研究员）拥有不同操作权限

### 1.3 用户角色

| 角色 | 职责 | 权限范围 |
|------|------|---------|
| **Admin（系统管理员）** | 系统维护、用户管理、全局配置 | 所有功能 |
| **Lab Manager（实验室管理员）** | 管理设备、房间、项目、审核使用记录 | 设备/房间/项目/使用记录管理 |
| **Researcher（研究员）** | 使用设备、记录数据、查看仪表盘 | 查看、使用设备、上传数据 |

## 二、功能设计

### 2.1 系统首页 —— 仪表盘

进入系统后，首先看到的是一个**可视化仪表盘**，一目了然地掌握实验室整体运行状态。

**展示内容：**

| 区域 | 内容 | 说明 |
|------|------|------|
| KPI 统计卡片 | 设备总数、当前使用中、项目总数、数据记录总数 | 顶部 4 个数字卡片 |
| 设备利用率图 | 柱状图，横轴为设备名称，纵轴为本月使用时长 | 一眼看出哪些设备最忙/最闲 |
| 使用趋势图 | 折线图，近 30 天每天的设备使用总时长 | 观察使用趋势，发现高峰/低谷 |
| 项目数据分布 | 饼图，各项目产生的数据记录数量占比 | 了解各课题的数据产出情况 |
| 最近活动 | 时间线，最近 10 条使用/数据操作记录 | 快速了解最近发生了什么 |

### 2.2 实验室管理 —— 楼层与房间

实验室采用层级结构管理：**实验室 → 楼层 → 房间**

**功能点：**

- 以卡片形式展示每个实验室，点击进入查看楼层
- 每个房间用卡片展示，显示房间号、名称、面积
- 支持新增、编辑、删除实验室/楼层/房间
- 房间内可查看关联的设备列表

**操作示例：**

```
Main Research Building
├── 1F
│   ├── Room 101 - Chemistry Lab (80m²) → UV光谱仪、离心机
│   └── Room 102 - Physics Lab (65m²) → [空]
└── 2F
    └── Room 201 - Biology Lab (90m²) → 扫描电镜
```

### 2.3 设备管理

设备是系统的核心管理对象，每台设备有详细的信息和状态追踪。

**设备信息：**

- 名称、型号、序列号（唯一标识）
- 所属房间（位置信息）
- 设备分类：光谱仪 / 显微镜 / 离心机 / 分析仪 / 其他
- 状态：可用 (available) / 使用中 (in_use) / 维护中 (maintenance) / 已退役 (retired)
- 制造商、购买日期、维护记录

**功能点：**

- 设备列表：表格展示，支持按房间、状态、类型、关键词筛选
- 设备详情页：基本信息 + 使用历史 + 关联数据记录
- 快捷操作：在设备详情页可一键"登录设备"或"登出设备"
- 状态联动：登录设备时自动变为"使用中"，登出后自动恢复"可用"

**设备详情页效果：**

```

├────────────────────────┬────────────────────────┤
│ Basic Information      │ Quick Actions          │
│ ─────────────────────  │ ─────────────────────  │
│ Name: UV-2600          │ [Login to Equipment]   │
│ Model: UV-2600         │ [Logout]               │
│ Serial: SN-001         │                        │
│ Status: ● Available    │                        │
│ Room: 101 - Chemistry  │                        │
├────────────────────────┴────────────────────────┤

│ ─────────────────────────────────────────────── │

│ Li      │ PROJ-002   │ 14:00     │ -       │ …  │
└─────────────────────────────────────────────────┘
```

### 2.4 项目管理

每个研究课题对应一个项目，项目可分配多名研究人员。

**项目信息：**

- 项目名称、编号（如 PROJ-2026-001）
- 项目状态：进行中 (active) / 暂停 (paused) / 已完成 (completed)
- 负责人、起止日期
- 项目描述

**功能点：**

- 项目列表：展示项目名称、编号、状态、负责人、成员数
- 项目详情页：基本信息 + 成员管理 + 关联使用记录 + 关联数据记录
- 成员管理：添加/移除研究人员，分配角色（lead / member）
- 数据归属：项目下可查看所有关联的实验数据

### 2.5 使用记录（核心功能）

这是系统的**核心业务**——追踪每台设备的使用行为。

**工作流程：**

```
研究员到达实验室
    ↓
选择设备 → 点击"登录设备" (POST /usage/start)
    ↓
系统记录：谁、什么时间、用了哪台设备、用于哪个课题
    ↓
设备状态自动变为 "使用中"
    ↓
使用结束 → 点击"登出设备" (PUT /usage/{id}/end)
    ↓
系统记录登出时间，自动计算使用时长
    ↓
设备状态自动恢复为 "可用"
```

**业务规则：**

- 每台设备同一时间只能有一个活跃的使用记录（防止重复登录）
- 登录/登出自动联动设备状态
- 使用记录关联到具体项目（可选，支持临时使用）

**使用记录列表：**

| 设备 | 使用人 | 课题 | 登录时间 | 登出时间 | 时长 | 状态 |
|------|--------|------|---------|---------|------|------|
| UV-2600 | 张伟 | 纳米材料表征 | 09:00 | 11:30 | 150 min | 已完成 |
| 5425R离心机 | 李明 | 蛋白质纯化 | 14:00 | - | - | 使用中 |
| SU5000电镜 | 王芳 | 细胞成像 | 08:30 | 12:00 | 210 min | 已完成 |

### 2.6 数据管理

实验设备产生的数据需要被记录、归档和追踪。

**数据记录信息：**

- 数据类型（measurement / spectrum / image / other）
- 文件名、文件大小
- 关联的设备、项目、使用会话、产生者
- 自定义元数据（JSON 格式，灵活存储）

**功能点：**

- 数据列表：展示所有数据记录，支持按设备、项目、用户筛选
- 文件上传：选择文件 + 填写元数据，上传到服务器
- 文件下载：点击即可下载原始数据文件
- 数据归属：每条数据记录自动关联到设备、项目和研究人员

### 2.7 用户管理（管理员功能）

系统管理员可管理所有用户账号。

**功能点：**

- 用户列表：展示用户名、姓名、邮箱、角色、状态
- 新增用户：设置用户名、密码、姓名、角色
- 编辑用户：修改信息、重置密码、调整角色
- 停用用户：软删除（标记为不活跃）

### 2.8 认证与权限

**登录流程：**

1. 用户输入用户名和密码
2. 后端验证，返回 JWT Token
3. 前端存储 Token，后续请求自动携带
4. Token 过期自动跳转登录页

**权限控制：**

| 操作 | Admin | Lab Manager | Researcher |
|------|-------|-------------|------------|
| 查看仪表盘 | ✅ | ✅ | ✅ |
| 管理实验室/楼层/房间 | ✅ | ✅ | ❌ |
| 管理设备 | ✅ | ✅ | ❌ |
| 管理项目 | ✅ | ✅ | ❌ |
| 使用设备（登录/登出） | ✅ | ✅ | ✅ |
| 上传数据 | ✅ | ✅ | ✅ |
| 查看使用记录 | ✅ | ✅ | ✅ |
| 管理用户 | ✅ | ❌ | ❌ |

### 4.2 数据库设计

共 9 张核心表：

```
users              ← 用户信息（账号、密码、角色）
labs               ← 实验室
floors             ← 楼层（关联实验室）
rooms              ← 房间（关联楼层）
equipment          ← 设备（关联房间）
projects           ← 项目/课题
project_members    ← 项目成员（多对多关联）
usage_records      ← 使用记录（关联设备、用户、项目）
data_records       ← 数据记录（关联设备、用户、项目、使用会话）
```

**层级关系图：**

```
Labs ──1:N── Floors ──1:N── Rooms ──1:N── Equipment
                                          │
Users ──N:M── Projects                     │
  │               │                        │
  └──── UsageRecords ─────────────────────┘
  └──── DataRecords  ─────────────────────┘
```

### 4.3 API 设计

所有接口前缀 `/api/v1`，统一响应格式：

```json
{
  "code": 200,
  "message": "success",
  "data": { ... }
}
```

**主要接口清单：**

| 模块 | 接口 | 方法 | 说明 |

| 认证 | /auth/login | POST | 登录获取 Token |

| 用户 | /users | GET/POST/PUT/DELETE | 用户 CRUD |
| 实验室 | /labs, /labs/{id}/floors, /floors/{id}/rooms | GET/POST | 层级管理 |

## 五、关键设计决策

- **UUID 主键**：防止 ID 枚举，便于未来数据迁移
- **Flyway 而非 ddl-auto**：生产安全，版本化迁移，可回溯
- **本地文件存储**：50人规模足够，file_path 字段便于未来迁移到对象存储
- **单体架构**：小规模下最简单，代码分层清晰可后期拆分
- **ECharts**：中文文档好，图表类型丰富，vue-echarts 集成方便
- **JWT 无状态认证**：前后端分离 SPA 的标准方案，无需服务端 Session
